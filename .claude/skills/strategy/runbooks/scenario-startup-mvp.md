# 操作手册：创业 MVP 构建

> **模式**：NEXUS-Sprint | **时长**：4-6 周 | **智能体**：18-22

---

## 场景

你要做一个创业 MVP——一个需要快速验证产品市场契合度的新产品。速度很重要，但质量也不能丢。目标是 4-6 周内从想法到有真实用户的上线产品。

## 智能体阵容

### 核心团队（始终活跃）
| 智能体 | 角色 |
|--------|------|
| 智能体编排者 | 流水线控制器 |
| 高级项目经理 | 需求转任务 |
| Sprint 排序师 | 待办管理 |
| UX 架构师 | 技术基础 |
| 前端开发者 | UI 实现 |
| 后端架构师 | API 和数据库 |
| DevOps 自动化师 | CI/CD 和部署 |
| 证据收集者 | 每个任务都 QA |
| 现实检验者 | 最终质量门禁 |

### 增长团队（第 3 周+ 激活）
| 智能体 | 角色 |
|--------|------|
| 增长黑客 | 获客策略 |
| 内容创作者 | 上线内容 |
| 社交媒体策略师 | 社交媒体活动 |

### 支持团队（按需）
| 智能体 | 角色 |
|--------|------|
| 品牌守护者 | 品牌识别 |
| 数据分析师 | 指标和仪表盘 |
| 快速原型师 | 快速验证实验 |
| AI 工程师 | 产品包含 AI 功能时 |
| 性能基准师 | 上线前压力测试 |
| 基础设施运维师 | 生产环境搭建 |

## 逐周执行

### 第 1 周：发现 + 架构（第 0 阶段 + 第 1 阶段压缩版）

```
第 1-2 天：压缩版发现
├── 趋势研究员 → 快速竞品扫描（1 天，不用出完整报告）
├── UX 架构师 → 核心用户流程线框图
└── 高级项目经理 → 需求转任务清单

第 3-4 天：架构
├── UX 架构师 → CSS 设计系统 + 组件架构
├── 后端架构师 → 系统架构 + 数据库 Schema
├── 品牌守护者 → 快速品牌基础（颜色、字体、语调）
└── Sprint 排序师 → RICE 评分的待办列表 + Sprint 计划

第 5 天：基础搭建
├── DevOps 自动化师 → CI/CD 流水线 + 环境
├── 前端开发者 → 项目脚手架
├── 后端架构师 → 数据库 + API 骨架
└── 质量门禁：架构包通过
```

### 第 2-3 周：核心构建（第 2 阶段 + 第 3 阶段）

```
Sprint 1（第 2 周）：
├── 智能体编排者管理开发-测试循环
├── 前端开发者 → 核心 UI（认证、主视图、导航）
├── 后端架构师 → 核心 API（认证、CRUD、业务逻辑）
├── 证据收集者 → 每个完成的任务都 QA
├── AI 工程师 → ML 功能（如适用）
└── 周末 Sprint 回顾

Sprint 2（第 3 周）：
├── 继续跑开发-测试循环做剩余功能
├── 增长黑客 → 设计病毒式传播 + 推荐系统
├── 内容创作者 → 开始制作上线内容
├── 数据分析师 → 搭建数据追踪和仪表盘
└── 周末 Sprint 回顾
```

### 第 4 周：打磨 + 加固（第 4 阶段）

```
第 1-2 天：质量冲刺
├── 证据收集者 → 全量截图套件
├── 性能基准师 → 压力测试
├── 前端开发者 → 修 QA 问题
├── 后端架构师 → 修 API 问题
└── 品牌守护者 → 品牌一致性审计

第 3-4 天：现实检查
├── 现实检验者 → 最终集成测试
├── 基础设施运维师 → 生产就绪检查
└── DevOps 自动化师 → 生产部署准备

第 5 天：门禁决策
├── 现实检验者判定
├── 如果"需要改进"：快速修复循环（2-3 天）
├── 如果"就绪"：开始上线
└── 高管摘要生成器 → 利益相关方简报
```

### 第 5-6 周：上线 + 增长（第 5 阶段）

```
第 5 周：上线
├── DevOps 自动化师 → 生产部署
├── 增长黑客 → 激活获客渠道
├── 内容创作者 → 发布上线内容
├── 社交媒体策略师 → 跨平台活动
├── 数据分析师 → 实时监控
└── 客服响应者 → 用户支持上线

第 6 周：优化
├── 增长黑客 → 分析和优化渠道
├── 反馈分析师 → 收集早期用户反馈
├── 实验追踪员 → 启动 A/B 测试
├── 数据分析师 → 第一周分析
└── Sprint 排序师 → 规划迭代 Sprint
```

## 关键决策点

| 决策点 | 什么时候 | 谁来决 |
|--------|---------|--------|
| 概念继续/终止 | 第 2 天结束 | 工作室制片人 |
| 架构通过 | 第 4 天结束 | 高级项目经理 |
| MVP 功能范围 | Sprint 规划时 | Sprint 排序师 |
| 生产就绪 | 第 4 周第 5 天 | 现实检验者 |
| 上线时机 | 现实检验者给出"就绪"后 | 工作室制片人 |

## 成功标准

| 指标 | 目标 |
|------|------|
| 上线产品的时间 | <= 6 周 |
| 核心功能完成度 | MVP 范围的 100% |
| 首批用户入驻 | 上线 48 小时内 |
| 系统可用性 | 第一周 > 99% |
| 用户反馈收集 | 前 2 周 >= 50 份 |

## 常见坑和怎么避

| 坑 | 怎么避 |
|----|--------|
| 构建阶段范围蔓延 | Sprint 排序师用 MoSCoW 管控——"Won't"就是不做 |
| 过度工程化追求扩展性 | 用快速原型师心态——先验证，再考虑扩展 |
| 为了速度跳过 QA | 证据收集者每个任务都跑——没有例外 |
| 没有监控就上线 | 基础设施运维师在第 1 周就搭好监控 |
| 没有反馈收集机制 | 数据追踪 + 反馈收集在 Sprint 1 就做进去 |
