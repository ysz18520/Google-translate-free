## 四、已修复问题（2026-05-21 — B 方案 + 实时翻译兜底）

| 优先级 | 问题 | 修复内容 | 文件 |
|--------|------|---------|------|
| 🔴 P0 | **body_html 只翻译标题，正文内容未翻译** | 根本原因：批量翻译把 `body_html` 当作一个整体字段存储，但前端 `applyTranslations` 按**文本节点粒度**替换，两者无法匹配。B 方案实现：① 后端新增 `extractTextSegmentsFromHTML()` 从 body_html 中提取纯文本片段（模拟浏览器文本节点拆分）；② `runTranslationJob` 为每个片段创建独立的 `body_html:hash` 字段记录；③ 引入 job 级翻译缓存避免重复调用百度 API；④ 资源级批量查询减少数据库往返 | `src/routes/translate.js` |
| 🔴 P0 | **前端文本匹配失败（`&nbsp;`、空白不一致）** | `applyTranslations` 增加归一化处理：将 ` `（`&nbsp;` 渲染结果）替换为普通空格，合并连续空白为单个空格，再 trim 匹配。替换时保留原始前导/尾随空白，不破坏 HTML 格式 | `public/widget.js` |
| 🔴 P0 | **theme 层面文本未翻译（按钮、导航、UI）** | 根本原因：B 方案只翻译了 product/page 的 body_html，但 Shopify 店铺大部分 UI 文本（Add to cart、Buy it now、Home、Catalog、Related products 等）在 theme 的 sections 中。修复：前端增加**实时翻译兜底**——`applyTranslations` 返回未命中缓存的文本列表，通过 `fetchRealtimeTranslations()` 批量发送到 `/api/translate/batch` 实时翻译，结果存入 `pageTranslations` 缓存并再次应用。覆盖场景：① 初始页面加载；② 语言切换重建；③ MutationObserver 动态内容加载 | `public/widget.js` |
| 🟡 P1 | **Docker 构建问题** | ① `npm ci` 改为 `npm install`（无 package-lock.json）；② Prisma schema 增加 `binaryTargets = ["native", "linux-musl-openssl-3.0.x"]` 解决 Alpine Linux 引擎缺失；③ `shopify-api.js` 增加 `require('@shopify/shopify-api/adapters/node')` 解决 SDK v11 适配器错误 | `Dockerfile`、`prisma/schema.prisma`、`src/utils/shopify-api.js` |
| 🟡 P1 | **widget.js 按钮不显示** | 根本原因：`saveOriginalSnapshot()` 保存快照前移除 widget DOM，但保存后未重新创建。修复：快照保存后调用 `createStyles()` + `createWidget()` 重新渲染 | `public/widget.js` |
| 🟡 P1 | **widget.js 版本号缓存** | ScriptTag URL 使用 `?v=${widgetVersion}`（从 package.json 读取），但 `app.js` 中 `/translator-widget.js` 路由已增加 `Cache-Control: no-cache` 响应头，确保浏览器始终获取最新代码 | `src/app.js` |

| 🟡 P1 | **管理后台无 Theme 翻译入口** | **D-UI：设置页新增"翻译 Theme"按钮**，支持进度轮询 | `public/translator/index.html` |

### D 方案覆盖范围

| 层级 | 覆盖内容 | 实现方式 | 预估覆盖率 |
|------|---------|---------|-----------|
| **Layer 1：Locales 文件** | Theme 默认文本（按钮、标签、aria 等） | 读取 `locales/en.default.json` 递归提取 | **70~80%** |
| **Layer 2：Navigation 菜单** | 导航菜单名称 | 读取 `GET /linklists.json` | **+5~10%** |
| **Layer 3：运行时缓存** | 硬编码文本、动态内容 | 前端实时翻译 → 回写数据库 → 下次命中 | **+5~10%** |

### 架构说明

**核心设计：后端预翻译，前端只读缓存**

```
管理后台点击"翻译 Theme"
   ↓
后端读取 Theme locales + Navigation
   ↓
调用百度 API 翻译（一次性）
   ↓
写入 Translation 表（resourceType='theme'）
   ↓
用户打开店铺 / 切换语言
   ↓
前端 GET /api/widget/translations → 返回 theme 翻译
   ↓
applyTranslations 直接命中缓存（零延迟）
```

**兜底机制保留**：极少数遗漏文本走前端实时翻译 → 结果回写数据库 → 下次命中。

| 🔴 P0 | **ES/FR/ARA 切换语言卡死** | 第一次尝试：过滤脚本 type + try-catch + finally 保护，未解决。根本原因是 v4 架构（`innerHTML` 完全重建 + 脚本重执行）与 Shopify 主题 JS 存在不可调和的冲突——EN 切换正常是因为不走翻译分支，而 ES/FR/ARA 的翻译操作会触发主题 JS 对 DOM 变化的反应，与脚本重执行产生冲突。修复：**v5 架构重构**——彻底放弃 `innerHTML` 重建方案，改用 WeakMap 记录每个文本节点的原始文本，切换时只恢复文本节点并重新应用翻译，零脚本重执行、零 innerHTML 替换 | `public/widget.js` |

### v5 架构变更

| 项目 | v4（旧） | v5（新） |
|------|---------|---------|
| 状态保存 | `originalBodyHTML = document.body.innerHTML` | `originalTexts = new WeakMap(node -> textContent)` |
| 恢复原文 | `document.body.innerHTML = originalBodyHTML` + `reExecuteBodyScripts()` | `restoreOriginalTexts()` 遍历 WeakMap 恢复每个文本节点 |
| 翻译应用 | 重建后 `applyTranslations()` | 直接 `applyTranslations()`，无需重建 |
| 表单状态 | `saveFormStates()` / `restoreFormStates()` | 不再需要（DOM 不被销毁） |
| 脚本重执行 | `reExecuteBodyScripts()` 重新执行所有内联 JS | **完全移除** |
| Observer | 重建后启动，检测新节点 | 实时检测新节点并记录原始文本 |
